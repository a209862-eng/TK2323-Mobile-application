Lab Submissions — Lab 1 to Lab 5

The Build-Up

LAB 01 / UI

Foundations: Building the First Screens

Reflection This lab introduced me to Jetpack Compose's declarative approach to layout — a real shift from the XML-based UI I expected. Getting spacing, colour, and typography to feel intentional from the very first screen set the visual language I carried through the rest of the project.

LAB 02 / INTERACTION

Making It Respond

Reflection Adding state and click handling was where the app started to feel alive. Working through remember and mutableStateOf taught me how Compose recomposes the UI, which became essential later when wiring ViewModel state across multiple screens.

LAB 03 / MATERIAL DESIGN

Giving the App a Voice

Reflection Material 3 components — cards, buttons, text fields — gave Tech-Savvy a consistent design system instead of one-off styled boxes. This lab is where the green-and-dark theme that now runs through every screen of the final app was born.

LAB 04 / NAVIGATION

Connecting the Screens

Reflection Navigation Compose turned a handful of disconnected screens into an actual app. Defining routes as constants and passing a shared ViewModel through NavHost became the backbone structure that every later feature — Aduan, Berdekatan, Teduhan, Wira Bot — was built on top of.

LAB 05 / ROOM DATABASE

Making Data Permanent

Reflection This was the lab that changed everything underneath the hood. Moving aduan data from an in-memory list to a Room database with DAO, Entity, and Repository layers meant reports survive app restarts — a small change with a big trust implication for a civic-reporting app.

The Case Files

PROJECT 01 / LAB 4 + WORKING FEATURES

Tech-Savvy — Foundation Release

Jetpack Compose Navigation ViewModel SDG 16

Citizens often have no simple, organised way to report local issues — complaints are scattered, undocumented, and easy for institutions to lose track of. Project 1 built the first working version of Tech-Savvy: a multi-screen app for logging and tracking community complaints (aduan).

Lessons learned Sharing one ViewModel across five screens via Navigation Compose was the biggest unlock — it meant the aduan list, profile, and reward points stayed in sync everywhere. I also learned that good navigation structure early saves a lot of refactoring later, which mattered a lot in Project 2.

PROJECT 02 / LAB 5 + API + SENSORS + CLOUD

Tech-Savvy — Connected Release

Room Database Firebase Firestore GNews REST API GPS Sensor

Project 1's aduan list lived only on one device, with no shared community visibility, no live justice-related information, and no way to confirm exactly where an issue was reported. Project 2 connects Tech-Savvy to the outside world — locally, in the cloud, and on the map.

Lessons learned Wiring four pillars together — Room for personal records, Firestore for a shared community feed, a live REST API for SDG 16 news, and GPS for accurate location tagging — taught me how much "data persistence" actually means in practice: knowing which data is private, which is shared, and which is fetched fresh every time.

Reflection & Learning Journey

From First Screen to Connected App

Looking back from Lab 1 to Project 2, the biggest shift wasn't technical — it was how I started thinking about data. In Lab 1 I was just trying to get a button to look right. By Project 2, every decision came back to a question of trust: where does this data live, who can see it, and what happens if the network drops or a sensor is denied? Working through Room, Firebase, a live REST API, and GPS for an SDG 16 app made that question feel real rather than academic — a civic-reporting tool only works if people believe their report actually went somewhere and will be acted on. That's the lesson I'm carrying forward.

Resources & References

What This Was Built With

  • API GNews REST API — used to fetch live news articles related to justice, governance, and SDG 16 for the in-app news feed.
  • Cloud Firebase Firestore — real-time cloud database for the community aduan (complaint) feed shared across users.
  • Local DB Room (AndroidX) — local persistence layer for each user's personal aduan history.
  • Sensor Google Play Services Location (FusedLocationProvider) + Android Geocoder — GPS sensor integration for location-tagged reports.
  • UI Jetpack Compose / Material 3 — declarative UI toolkit and design system used throughout the app.
  • AI Assistance Portions of this app's UI scaffolding, Firebase/REST API/GPS integration code, debugging guidance, and this e-Portfolio page were developed with assistance from Claude (Anthropic). All code was reviewed, tested, and understood before submission.
